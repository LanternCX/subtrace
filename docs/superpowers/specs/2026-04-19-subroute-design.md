# Subroute 设计说明

## 目标

构建一个仅支持 macOS 的命令行工具 `subroute`，输入一个 Base64 订阅地址后，自动拉取订阅、解析节点、执行入口连通性与去程路由探测，并输出一份便于人工阅读的 Markdown 报告。

## 范围

- 支持通过 `subroute [url]` 直接执行。
- 输入订阅为 Base64 文本，解码后按行解析节点。
- 第一版只要求从节点条目中提取最小测试信息：协议名、节点名、目标地址、端口。
- 仅测试本机到节点入口地址的去程路线。
- 路由探测以 macOS 系统命令为基础，保留每一跳的原始结果与三次延迟。
- 对每一跳中返回的 IP 补充 AS 和归属地信息。
- 生成 Markdown 报告，报告以原始证据为主，附带少量摘要。

## 不做的事情

- 不做规则分组与代理配置生成。
- 不做通过代理访问外部目标的验证。
- 不做自动评分与自动排序。
- 不承诺 Linux 或 Windows 兼容。
- 不自动安装系统探测工具。

## 主流程

1. 接收订阅地址。
2. 拉取订阅内容并做 Base64 解码。
3. 逐行解析节点条目。
4. 将可提取最小测试信息的节点加入测试队列。
5. 对每个节点执行基础连通性检查与去程路由探测。
6. 汇总探测中出现的有效 IP，查询 AS 与归属地信息。
7. 将节点明细、原始跳数、标注信息和异常情况整理为 Markdown 报告。

## 节点处理规则

- 只要某条节点能够提取出协议名、节点名、目标地址、端口，就纳入测试。
- 解析失败的条目不做猜测，记入“无法解析/无法测试”部分。
- 协议信息以订阅条目自身声明为准，不做真实握手验证。

## 探测策略

- `ping` 结果作为基础参考信息，不可用时也保留节点，不因此跳过报告。
- `traceroute` 结果作为核心证据来源。
- 跳数有几跳就记录几跳，保留每一跳的原始行文本。
- 每一跳保留原始三次延迟。
- 对每一跳里返回的每个有效 IP 做 AS 与归属地标注。
- 查询失败或限流时，报告中标记为未知，不影响整份报告生成。

## 报告结构

- 顶部总览：订阅地址、测试时间、总节点数、可测节点数、解析失败条目数。
- 节点小节：节点名、协议、入口地址、端口、基础摘要。
- 去程路由表：跳数、原始结果、对应 IP 的 AS 与归属地标注。
- 异常小节：无法解析、探测失败、归属查询失败等情况。

## 成功标准

- 对一个有效的 Base64 订阅地址执行后，能够稳定生成 Markdown 报告文件。
- 报告中能区分可测节点与不可测条目。
- 可测节点包含基础连通性信息与逐跳路由信息。
- 逐跳路由信息包含原始结果、延迟、AS 和归属地标注。
- 单个节点失败不会阻塞其他节点的处理。
